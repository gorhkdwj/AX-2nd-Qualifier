# 검증 계획 · AX해커톤 예선 2차 제출(기업 미정)

기능을 만들 때마다 영향 범위에 맞게 검증한다. 검증하지 못한 부분은 숨기지 않고 `미검증 범위`에 적는다. 실패한 테스트를 삭제·완화하지 않고 원인을 수정한다.

## 검증 항목
- 기업 선정 근거: 선정된 기업의 문제가 공개 자료로 실제 확인되는가 (URL 접속·내용 일치 확인)
- 문서-코드 정합성: `docs/requirements-contract.md`와 실제 플러그인 동작 일치 여부
- README-기능 일치: README에 쓴 기능이 실제로 동작하는지
- 제출물/배포물 구조: `submission.zip` 기준 — `src/.codex-plugin/plugin.json` 존재, 스킬/MCP/코드 중 1개 이상 존재, 루트에 `README.md`·`logs/` 존재
- 로그 무결성: `logs/` 내용이 훅으로 자동 저장된 원본 그대로이며 사후 편집이 없는가
- 질문 5문항 정합성: 답변 내용이 실제 로그·플러그인 코드와 모순되지 않는가

## 검증 방법과 도구
- 공개 자료 검증: 실제 URL 접속 및 내용 대조 (WebFetch 등)
- 제출 구조 검증: 폴더 구조 수동 점검(`ls`, `Glob`) + 체크리스트 대조
- 로그 무결성: `tools/save_log.py` 훅 동작 결과와 `logs/` 파일을 비교, 수동 수정 이력 없는지 확인

## 미검증 범위
- 현재 기업이 미확정 상태이므로 플러그인 기능·성능에 대한 검증은 아직 수행 대상이 아님.
- 심사위원의 실제 채점 기준(AI+심사자 평가)은 외부 프로세스이므로 이 문서로 사전 검증 불가.
