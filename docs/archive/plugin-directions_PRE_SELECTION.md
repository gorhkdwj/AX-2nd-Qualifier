## 0. 데이터 취득·검증 정책 (모든 방향에 공통 적용)

과제 규정(공개·검증 가능한 근거, 심사자 재현 가능)과 법적 안전성을 함께 만족시키기 위해, 아래 원칙을 강제합니다.

- **검증(도구가 올바로 작동하는가)**: **합성(더미) 픽스처 우선.** 정답 라벨을 아는 통제된 입력(정상/경계/오류 케이스)을 직접 만들어 precision/recall을 측정한다. 실데이터보다 재현·엄밀성이 높다.
- **문제 실재성(근거)**: 이미 확보한 **공개 출처 인용**으로 대신한다(뉴스룸·판결·인터뷰·공식 도움말 등). 대량 데이터 수집 불필요.
- **현실성 점검**: 필요 시 **소수(3~5건)의 공개 샘플**만 출처와 함께 인용해 sanity check.
- **지식원(RAG 대상)**: 기업의 **공식 발행물**(도움말·정책)과 **공공 법령·고시**만 사용. 도움말은 전체 자동 미러링이 아니라 **소량 큐레이션 스냅샷 + 출처 링크**로 `docs/references/`에 보관.
- **입력 방식**: 플러그인이 남의 사이트를 자동으로 긁지 않는다. **사용자가 텍스트를 붙여넣거나(BYO), 사용자가 권한을 가진 자기 페이지를 지정**하는 방식으로 설계.
- **금지선**: **제3자 UGC(리뷰·커뮤니티) 대량 크롤링 금지.** (저작권·DB제작자권·부정경쟁방지법·개인정보보호법·이용약관 위반 위험. 특히 출제 3사는 데이터 부정취득 판례(마이리얼트립 v. 민다, 2025)가 있는 영역이라 더 민감.) 어떤 사이트든 robots.txt·이용약관을 준수한다.

> 요약: **더미로 정확성 검증 + 소량 공개 인용으로 실재성·현실성 보강. 지식원은 공식·공공 자료. 입력은 BYO. 제3자 리뷰 대량 수집은 하지 않는다.**

## 1. 무신사 (방향 2개 · 택일)

무신사에는 성격이 다른 두 방향이 있습니다. **문제 1(표기 검수기)**은 공개자료로 검증된 규정 준수 문제로 근거가 가장 단단하고, **문제 2(에이전트화 변환기)**는 무신사 테크리드 인터뷰(`docs/company-research_references/무신사 인터뷰.txt`)의 핵심 비전("AI 에이전트가 패션 질문 때 첫 번째로 쓰는 도구가 무신사이길", "동일 상품 두 번 등록/파편화")에 정면으로 맞는 방향입니다. 두 방향은 택일 후보이며, 근거 검증 강도(문제 1)와 인터뷰 주제 정합성(문제 2)의 트레이드오프가 있습니다(하단 비교표 참조).
> (기존 1-A와 1-B를 하나로 통합해 문제 1로. 두 모듈은 "소재 혼용률의 정확성"과 "필수 표시항목의 완비성"으로 사실상 하나의 표기 검수 도구. 기존 1-C(리뷰·뉴스 평판 모니터링)는 안전성 기준으로 제외 — 부기 B 참조.)

### 문제 1 · 상품정보 표기의 오류·누락 반복
- **근거(공개)**: 2024.12~2025.1 다운·캐시미어 7,968개 전수조사에서 42개 브랜드·165개 상품 오기재 적발(무신사 뉴스룸 2025-0123-2, 이투데이) → 2025.11~12 노스페이스 패딩 13개 스타일(28 SKU) 혼용률 오기 재발, 대행사 벌점 50점·선제 전액 환불(연합뉴스TV 2025-11-28, 무신사 뉴스룸 2025-1202-2, 스카이데일리·산경투데이). 재발 방지책에도 1년 내 동일 유형 재발 `[검증됨/교차확인]`.
- **공개자료만으로 가능한 이유**: 점검 대상은 공개 상품 상세페이지(사용자/셀러가 자기 페이지를 입력), 판정 기준은 공개 규정 — 무신사가 도입한 "기타 섬유" 포괄표기 금지·15% 이내 세분화 표기, 국가기술표준원 섬유 표시기준, 전자상거래 **상품정보제공고시**(의류 필수 표시항목). 내부 검수 시스템 접근 불필요.
- **플러그인 방향 — `무신사 상품정보 표기 검수기`** (BYO 입력)
  - 입력: 상품 상세페이지 텍스트(붙여넣기) 또는 사용자가 지정한 자기 상품 URL, + (선택) 케어라벨 이미지
  - 처리(두 모듈):
    - ⓐ **필수항목 완비성 점검**(구 1-B): 카테고리별 상품정보제공고시 필수항목(소재·제조국·제조자·세탁방법·치수·A/S 등) 누락·공란·상호 모순 검출
    - ⓑ **소재 혼용률 심층 검증**(구 1-A): 혼용률 합계 100%·논리오류, 포괄표기 사용, 15% 세분화 위반, 텍스트 표기 ↔ 케어라벨 이미지(OCR) 불일치, 소재명 표준용어 매핑
  - 출력: 위반·누락 항목 목록, 근거 규정 인용, 수정 문구 초안, 승인/보류 판정
  - 형태: SKILL(판정 규칙) + 표기 규칙·필수항목·표준용어 사전(정적 데이터) + (선택) 이미지 OCR 코드
  - 검증: 정상/누락/혼용률오류/포괄표기 등 더미 픽스처로 정확성 측정 + 공개된 실제 사례(노스페이스 보도 내용) 소수로 sanity check

### 문제 2 · 상품 데이터가 에이전트가 이해할 구조가 아님 (에이전트-first 전제 미충족 + 파편화)
- **근거**: 무신사 테크리드 인터뷰에서 "AI 에이전트가 패션을 물을 때 **첫 번째로 쓰는 도구가 무신사이길**", "여러 플랫폼(무신사·솔드아웃 등)이 많아질수록 기술이 파편화되고 **동일 상품을 두 번 등록해야 하는 불편함**"을 직접 언급 `[인터뷰 신호 — 전사본이라 제출 인용 시 원본 공개영상 URL·정확한 인용 확인 필요]`. 멀티플랫폼·글로벌 스토어 확장은 공개 사실 `[검증됨]`. 현재 상품 상세정보는 대체로 비정형 텍스트라 에이전트가 정확히 질의·필터하기 어렵다.
- **공개자료만으로 가능한 이유**: 입력은 **공개 상품 상세페이지 텍스트(BYO — 사용자가 붙여넣거나 자기 페이지 지정)**, 정규화 기준은 **공개 패션 카테고리 taxonomy·소재 표준용어**. 내부 카탈로그·판매 데이터 불필요.
- **플러그인 방향 — `무신사 상품 데이터 에이전트화(化) 변환기`** (BYO 입력)
  - 입력: 상품 상세페이지 텍스트(비정형: 제목·설명·소재·이미지 alt 등)
  - 처리: ① 정규 스키마 속성 추출(카테고리/서브카테고리, 핏·실루엣, 소재, 컬러, 계절, TPO·스타일 태그, 관리, 사이즈) ② 공개 표준 taxonomy 매핑(자유표기→표준용어) ③ 중복 등록 감지(같은 상품의 다른 표현을 속성 유사도로 판별) ④ 에이전트 질의 대응 descriptor 생성("여름 결혼식 하객룩" 류 자연어 의도 매칭)
  - 출력: 정규화된 구조화 JSON + 부족·모호 속성 지적 + 중복 의심 경고 + 에이전트 질의 예시(질의→필터→설명)
  - 형태: SKILL + 패션 taxonomy·소재 표준용어 사전(정적) + (선택) 유사도 판별 코드
  - 검증: 정답 속성이 라벨된 더미 상품설명 세트로 추출 정확도(속성별 precision/recall), 같은 상품 두 표현 쌍 더미로 중복 감지 정확도, 공개 상품페이지 소수로 sanity check

- **왜 이 변환기가 "에이전트가 첫 번째로 쓰는 도구"로 이어지나 (인과 사슬)**
  에이전트가 어떤 도구를 먼저 쓰는지는 "그 도구가 질문에 얼마나 정확·일관·설명 가능하게 답하느냐"로 정해지고, 그 답변력의 전제가 "상품 데이터가 에이전트가 이해할 구조인가"다. 변환기가 그 전제를 만든다.
  1. **기계가 읽을 속성 확보**: 비정형 텍스트 → 구조화 속성 → 에이전트가 필터·추론할 원재료 생성.
  2. **표준 taxonomy = 일관된 매칭**: "린넨/리넨/마"·"하객룩" 등을 표준화 → 정확·일관된 매칭(도구 선호의 핵심은 안정적 신뢰성).
  3. **질의 의도 대응 descriptor**: 자연어 의도와 상품 속성 사이 간극을 메워 답변 정확도↑.
  4. **설명 가능성**: 구조화 속성으로 "왜 맞는지" 설명(예: "린넨 혼방이라 통기성↑, 세미정장 핏이라 하객룩 적합") → 신뢰 → 반복 사용.
  5. **중복 제거·canonical 스키마**: 플랫폼 간 중복 없는 깔끔한 결과 + 무신사를 파편화된 사일로가 아닌 하나의 질의 가능한 카탈로그(원 코어 멀티 플랫폼)로 통합.
  6. **도구 선택 역학**: 에이전트는 (a)연결 가능(MCP) (b)안정적 답변 (c)구조화·설명 가능 데이터를 주는 도구를 선택 → 변환기가 (b)(c)를 결정적으로 끌어올림 → 무신사가 "그냥 잘 되는" 패션 도구가 되어 기본값으로 선택됨.
  - **경계(정직)**: 변환기 하나만으로 1번 도구가 되는 건 아니다. MCP/도구 인터페이스, 전 상품 커버리지, 랭킹 로직이 함께 필요하다. 다만 그 모든 것의 **전제(에이전트가 이해할 상품 데이터)**가 없으면 나머지가 성립하지 않으므로, 변환기는 **없으면 안 되는 기반 첫 레이어**다. 이 "좁고 명확한 문제 정의"가 인터뷰 평가기준(문제-해결 매칭·좁은 정의·임팩트)에 정확히 부합한다.
  - **시연(end-to-end 메커니즘)**: 공개 상품페이지 텍스트 입력 → 구조화 JSON 출력 → 샘플 에이전트 질의("여름 하객룩")를 그 JSON 위에서 필터·설명으로 답변 — 이 한 흐름으로 "비정형→에이전트가 답하는 도구"라는 인과를 시연(공개페이지+더미로 안전 재현).

### 무신사 문제 1 vs 문제 2 — 선택 기준
| 관점 | 문제 1 (표기 검수기) | 문제 2 (에이전트화 변환기) |
|---|---|---|
| 인터뷰 주제 정합성 | 낮음(규정 준수·백오피스) | **높음**(에이전트-first·파편화, 인터뷰 핵심 비전 직격) |
| 문제 근거의 검증 강도 | **높음**(노스페이스 사건 등 공개 검증) | 중간(인터뷰 신호 위주, 전사본 → 원본 확인 필요) |
| 공개데이터·안전성 | 안전(BYO+공개규정+더미) | 안전(BYO+공개 taxonomy+더미) |
| 임팩트 서사 | 환불 출혈·벌점·신뢰 훼손 | 에이전트 시대 검색 노출·중복작업 감소 |
| 평가 루브릭(문제-해결 매칭·임팩트·메커니즘) | 강 | 강 |
> 결론: 주제 정합성을 우선하면 문제 2, 근거 검증 강도를 우선하면 문제 1. 둘 다 안전·재현 가능하며 택일 후보다. (선정은 사용자 결정.)

## 2. 메디테라피 (방향 2개)

메디테라피에서 공개자료로 검증된 안전한 문제는 **화장품 광고 규정 리스크**와 **퍼포먼스 광고 대비 낮은 전환/콘텐츠**입니다. 판정 기준(식약처 지침)과 점검 대상(자사 공개 문구)이 공개라 크롤링이 필요 없습니다.
> (기존 2-C 글로벌 다국어 리뷰 트래커는 리뷰 대량 수집이 핵심이라 안전성 기준으로 제외 — 부기 B 참조. 2-B는 리뷰 의존을 제거하고 자사 공개페이지 기반으로 재정의.)

### 문제 1 · 화장품 표시·광고 규정 위반 리스크 (과대광고 등)
- **근거(공개)**: 고효능 성분 브랜드로서 광고 문구의 규제 리스크가 큼. 식약처 **화장품 표시·광고 관리 지침**(민원인안내서, 2025.8 게시)이 공개 규정으로 존재 `[검증됨]`. 자사몰 상세·광고 문구도 공개.
- **공개자료만으로 가능한 이유**: 판정 기준(식약처 지침·화장품법 금지표현)과 점검 대상(자사 공개 문구)이 모두 공개.
- **플러그인 방향 — `화장품 광고문구 리스크 점검기`** (BYO 입력)
  - 입력: 광고/상세페이지 문구(붙여넣기 또는 자사 URL), 제품 유형(일반/기능성)
  - 처리: 의약품 오인·질병 치료 암시·배타적/최상급 표현·미승인 효능 표방 등 금지 표현을 식약처 지침 기준으로 탐지 → 위험도 등급화
  - 출력: 위반 소지 문구·근거 지침 조항·대체 표현 제안
  - 형태: SKILL + 금지/주의 표현 규칙 사전(정적)
  - 검증: 금지표현 포함/정상 문구 더미 픽스처로 탐지 정확도 측정

### 문제 2 · 퍼포먼스 광고 → 구매전환 병목 (자사 페이지 기반)
- **근거(공개)**: 피처링 고객사례 인터뷰에 "광고 유입이 구매로 이어지지 못해 ROAS 개선 돌파구가 절실", "진정성 있는 브랜드 콘텐츠 부족"이 담당자 발언으로 확인됨 `[검증됨]`.
- **공개자료만으로 가능한 이유**: 진단 대상이 **자사 공개 랜딩/상세페이지·제품 USP**(사용자가 자기 페이지 입력). 실제 캠페인 성과(비공개) 없이 "메시지·구조" 관점 진단·제안이 가능. 제3자 리뷰 수집에 의존하지 않음.
- **플러그인 방향 — `전환 병목 진단 & 콘텐츠 코치`** (BYO 입력)
  - 입력: 자사 랜딩/상세페이지 문구, 제품 USP (+ 선택: 사용자가 제공한 소수의 공개 리뷰 인용)
  - 처리: 전환 관점 체크(가치제안 명확성·신뢰요소·CTA·구매장벽) → 병목 진단 → 개선 카피 초안 제안
  - 출력: 병목 진단표·개선 우선순위·카피 초안
  - 형태: SKILL. (선택) 문구 구조 분석 코드
  - 검증: 병목 유형이 심어진 더미 랜딩 문구 세트로 진단 정확성 확인

## 3. 마이리얼트립 (방향 3개)

마이리얼트립에서 공개자료로 검증된 안전한 문제는 **예약·취소·환불 문의의 구조적 복잡성**과 **공개 도움말/정책 콘텐츠 자체의 정합성**입니다. 지식원이 전부 기업 공식 도움말(공개)이라 "구조 개선·제안"형에 특히 적합합니다. (지식원은 소량 큐레이션 스냅샷 + 출처로 구성, 전체 자동 크롤링 없이.)

### 문제 1 · 예약/취소/환불 문의의 분류·정확한 규정 안내 부담
- **근거(공개)**: 도움말이 예약 33개·취소환불 7개·상품문의 26개로 세분, 24시간 채팅 상담 운영(help.myrealtrip.com) `[검증됨]`.
- **공개자료만으로 가능한 이유**: 답변 근거가 공식 도움말 전체(공개). 이를 지식원으로 한 RAG로 충분.
- **플러그인 방향 — `여행 문의 triage & 답변 초안기`**
  - 입력: 고객 문의 문장(+선택: 예약 유형/시점)
  - 처리: 문의 유형 분류 → 관련 공식 도움말 검색·인용 → 답변 초안 + 추가 확인정보 목록
  - 출력: 문의 유형·근거 FAQ 링크·답변 초안·필요 확인정보
  - 형태: SKILL + 공식 도움말 큐레이션 코퍼스(출처 포함) + 검색 로직
  - 검증: 더미 문의 세트에 대해 유형 분류 정확도·근거 링크 포함 여부 측정

### 문제 2 · 취소·환불 규정 복잡성(항공사별·사유별) 오안내
- **근거(공개)**: 항공사별 24시간 무료취소 정책 문서, "부득이한 사유" 취소 가이드(증빙 필요·영업일 7일), 예약확정 SLA(영업일 2일) 등이 공개 도움말로 존재 `[검증됨]`.
- **공개자료만으로 가능한 이유**: 적용 규정·필요 증빙·예상 소요일 산정에 필요한 정책 문서가 전부 공개.
- **플러그인 방향 — `취소·환불 규정 네비게이터`**
  - 입력: 예약 유형(항공/숙박/투어)·결제 시점·취소 사유
  - 처리: 조건 분기(24시간 이내 여부, 부득이한 사유 여부 등)로 적용 규정·필요 증빙·예상 소요일·수수료 여부 결정
  - 출력: 적용 규정 요약·필요 서류 체크리스트·예상 처리기간·근거 링크
  - 형태: SKILL + 정책 규칙(공개 문서 기반 결정 트리)
  - 검증: 조건별 시나리오 더미 입력으로 규정 매핑 정확성 확인

### 문제 3 · 공개 도움말 콘텐츠의 정합성·최신성 결함 (구조 개선·제안형)
- **근거(공개)**: 24시간 무료취소 정책 문서가 실존하나 **본문 텍스트가 사실상 비어 있음(이미지·첨부 위주)**을 직접 확인 `[검증됨]`.
- **공개자료만으로 가능한 이유**: 점검 대상이 공개 도움말 그 자체. 외부 데이터 불필요.
- **플러그인 방향 — `도움말 정합성·최신성 감사기`**
  - 입력: 공식 도움말 문서 모음(URL/텍스트)
  - 처리: 빈 문서·텍스트 부재 검출, 문서 간 상호 모순(예: 취소 소요일 불일치), 오래된 정책·날짜 미표기, 카테고리 커버리지 공백 도출
  - 출력: 결함 목록·중요도·개선 제안(우선순위)
  - 형태: SKILL + 문서 파싱/비교 코드. 심사자가 공개 도움말로 즉시 재현 가능

## 부기 A · 선정 시 유의
- 위 방향은 모두 공개 소스만으로 시연·검증 가능하도록 설계했습니다. 구현 시 각 플러그인의 지식원(규정·표준·도움말)을 출처와 함께 `docs/references/`에 보관하고, `docs/requirements-contract.md`에 입력·판정 기준을 고정하십시오.
- 검증 데이터 정책은 `docs/validation-plan.md`에 명문화되어 있습니다(더미 우선 + 소량 공개 인용, 크롤링 금지선).
- 남은 안전 방향 요약: 무신사 2개 택일(표기 검수기 / 에이전트화 변환기), 메디테라피 2개(광고규정 점검·전환 코치), 마이리얼트립 3개(문의 triage·규정 네비·도움말 감사).
- 확정 시 그 사의 방향 중 **가장 작은 성공 단위 1개**를 MVP로 잡아 진행 권장(마감 2026-07-06 밤).

## 부기 B · 안전성 기준으로 제외한 방향 (기록 보존)

- **무신사 (구 1-C) 공개 리뷰·뉴스 기반 품질·평판 리스크 조기경보**: 본질이 외부 리뷰·뉴스 대량 모니터링 → 제외.
- **메디테라피 (구 2-C) 글로벌 다국어 공개 리뷰 이슈 트래커**: 존재 이유가 아마존·쇼피 등 리뷰 대량 수집 → 제외.
