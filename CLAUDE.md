# CLAUDE.md / AGENTS.md · AX해커톤 예선 2차 제출(기업 미정) 작업 헌법

이 문서는 이 프로젝트에서 AI 협업 에이전트(Claude, Codex 등)가 **매번 지시하지 않아도 항상 따라야 하는 기준**이다. 사용자와 에이전트 모두 이 규칙을 기준으로 일한다. 규칙이 바뀌면 `CLAUDE.md`와 `AGENTS.md`를 함께 갱신하고 Decisionlog에 이유를 남긴다.

## 0.1 에이전트 지침 파일 동기화

- `CLAUDE.md`와 `AGENTS.md`는 같은 프로젝트 작업 헌법을 담는 쌍둥이 문서다. `CLAUDE.md`는 Claude 계열 도구가, `AGENTS.md`는 Codex 계열 도구가 우선 참조할 수 있도록 둔다.
- 두 파일의 본문은 항상 동일하게 유지한다. 에이전트별로 다른 규칙이 꼭 필요하면 문서를 분기하지 말고, 같은 문서 안에 "Claude 전용" 또는 "Codex 전용"이라고 명시한다.
- 두 파일 중 하나를 수정해야 하면 작업 전에 두 파일을 모두 읽고 차이를 확인한다.
- 한쪽만 변경되어 있으면 변경 내용을 다른 쪽에도 즉시 반영한다.
- 양쪽 모두 변경되어 있고 내용이 충돌하지 않으면 병합한 뒤 두 파일에 동일하게 반영한다.
- 양쪽 모두 변경되어 있고 지침 의미가 충돌하면 임의로 한쪽을 버리지 않는다. 더 엄격하고 안전한 규칙을 임시 기준으로 삼되, 사용자 확인이 필요한 항목은 `확인 필요`로 표시하고 Worklog 또는 Decisionlog에 남긴다.
- 수정 후에는 두 파일의 내용 또는 해시가 동일한지 확인한다.
- 지침 의미가 바뀐 경우 Decisionlog에 이유와 영향을 기록하고, 단순 문구 정리나 동기화 작업은 Worklog에 기록한다.

## 1. 대화와 보고 방식

- 결론을 먼저, 근거·세부는 뒤에.
- 기술 용어는 처음 등장할 때 한 줄로 설명한다.
- 불확실하면 단정하지 말고 `확인 필요`로 표기한다.
- 실행/확인하지 않은 것을 완료라고 말하지 않는다.

## 2. 작업 시작 전 확인

- 현재 폴더 구조와 기존 파일을 먼저 확인한다.
- 기존 파일·변경사항을 함부로 덮어쓰지 않는다.
- 바로 구현하지 말고, 영향 범위와 계획을 먼저 정리한다.

## 3. 파일 구조 원칙

- `src/` 실제 제출될 Codex 플러그인 코드(`.codex-plugin/plugin.json`, `skills/`, `.mcp.json` 등) — **submission.zip에 그대로 포함되는 영역**.
- `README.md`(루트) 최종 제출용 설명 문서 — **submission.zip에 포함**.
- `logs/` AI 대화 로그(claude-code, codex 등) — **submission.zip에 포함, 원본 그대로**.
- `docs/` 기획·리서치·근거·검증 등 내부 작업 문서(제출물 아님, 참고용). `docs/technical_references/`에 Codex 플러그인 공식 문서 사본이 이미 준비되어 있음(Plugins, Build_plugins, Agent_Skills).
- `tools/` 개발 보조 스크립트(예: `save_log.py`) — 제출물 아님.
- `tests/` 플러그인 동작 검증용 스크립트/케이스 — 필요 시 사용, 제출물 아님.
- `out/` 로컬 실행 결과·임시 산출물(Git 제외) — 제출물 아님.
- 필요 없는 폴더는 만들지 않고 이유를 남긴다.

## 4. 구현 원칙

## 10. Git 원칙

- 원격 저장소: `https://github.com/gorhkdwj/AX-2nd-Qualifier.git` (`origin/main`). 이미 초기화·연결되어 있다.
- **모든 작업(주요 사용자 요청)이 끝날 때마다 즉시 commit → push 한다.** 순서: 변경 파일 확인(`git status`) → 불필요·비밀 파일 제외 확인 → 필요한 검증 실행 → commit → push. 사용자가 별도로 요청하지 않아도 매 작업 종료 시 자동으로 수행한다.
- 로컬 실행 결과, 비밀정보, 임시 산출물(`out/`), **그리고 `logs/`(AI 대화 로그)는 Git에 올리지 않는다.** `logs/`는 `.gitignore`로 제외하며 로컬에서만 관리한다.
- **단, `logs/`는 과제 제출물(`submission.zip`)에는 반드시 포함해야 한다.** 즉 git에는 올리지 않되, 최종 제출 시점에 로컬 `logs/`를 zip에 직접 담는다. (git 저장소 ≠ 제출물)
- commit·push 전에는 반드시 스테이징 대상에 `logs/`나 비밀정보가 섞이지 않았는지 `git status`로 확인한다.

## 11. 로그 기록 규칙

### Worklog.md — 주요 요청마다

모든 주요 사용자 요청이 끝날 때마다 기록한다. 설명 작업이라도 방향·구조·판단에 영향을 주면 기록한다.

```
### W-00N · 작업 제목
**요청** / **수행 작업** / **변경 파일** / **검증**(미실행 시 사유) / **판단 근거** / **결과**(완료·남은 작업)
```

### Decisionlog.md — 방향을 바꾸는 결정만

대상: 문제 선정(기업 선정 포함), 기술 스택 선택, 구현 범위 확정·축소, 데이터 처리 방식, 보안·개인정보 처리, 테스트·검증 방식, 배포·제출·공유 방식, 큰 구조 변경. 사소한 수정은 Worklog만.

```
### D-00N · 결정 제목
**상황** / **검토한 선택지** / **결정** / **근거** / **영향** / **재검토 조건**
```

### Troubleshootinglog.md — 실제 문제 발생 시

실제 오류·실패·환경 문제·검증 실패·설계 충돌이 발생하면 기록한다. 같은 문제가 반복되면 새 T-ID를 만들기 전에 기존 T-ID를 먼저 확인한다.

```
### T-00N · 문제 제목
**발생 상황** / **증상** / **확인된 원인**(+불확실점) / **조치**(+최종 해결) / **재발 방지**
```
